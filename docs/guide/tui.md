# The TUI

`ft tui` opens a tabbed full-screen interface built on `ratatui`.
Every chord you press is a named command in a registry shared with the
CLI, so `ft commands list`, the generated reference at
[docs/keybindings.md](../keybindings.md), the `?` overlay inside the
TUI, and the headless `ft do` command all read from the same source.

This chapter is the tour. The detailed reference for each tab's
behavior lives in the corresponding chapter ([tasks.md](tasks.md),
[notes.md](notes.md), [timeblocks.md](timeblocks.md),
[graph.md](graph.md)). The Command/Keymap model itself is in
[docs/commands.md](../commands.md).

## Launching

```sh
ft tui
ft tui --vault /path/to/other/vault
```

The TUI inherits vault discovery from the same precedence as every
other subcommand (see
[vault-and-config.md](vault-and-config.md#vault-discovery-in-order)),
so just running `ft tui` from any directory inside a vault works.

## The five tabs

Across the top: **Graph**, **Tasks**, **Notes**, **Timeblocks**, and
**Journal**. Each has its own keymap; the active tab decides which
chords are bound. The Graph tab is the most powerful — most "what's
related to what" workflows start there.

| Index | Tab        | What it's for                                                                  |
|-------|------------|--------------------------------------------------------------------------------|
| `1`   | Graph      | Interactive DSL-driven tree of notes, directories, and link / embed edges.     |
| `2`   | Tasks      | Live task triage. CLI presets become sidebar views; mutations one chord away.  |
| `3`   | Notes      | Fuzzy note picker, section operations, template creation, quick capture.       |
| `4`   | Timeblocks | Day-planner editor with split today + tomorrow view.                           |
| `5`   | Journal    | Reverse-chronological feed of paragraph-level mentions of a note.              |

## Global chords

These are always bound, regardless of which tab is active:

| Key             | Action                                                |
|-----------------|-------------------------------------------------------|
| `Tab` / `Shift+Tab` | next / previous tab                              |
| `1`–`9`         | jump to tab N (1-indexed)                            |
| `?`             | open the keymap help overlay for the active context  |
| `q` / `Ctrl+C`  | quit                                                  |
| `g` then `s`    | run `ft git sync` on a background thread (the `g`-leader chord) |

The git-leader is a two-stroke chord: `g` arms it (with a hint in the
status bar), then `s` triggers the sync. Any other key cancels.

## The `?` overlay

Press `?` to see every chord bound in the active context (tab,
modal, or App-global), grouped by category. The overlay is generated
from the registry, so it never drifts from what's actually wired up.

For the same information outside the TUI, run:

```sh
ft commands list                          # table grouped by scope
ft commands list --scope tab/tasks        # only Tasks-tab commands
ft commands list --format ndjson          # script-friendly
```

The committed reference at [docs/keybindings.md](../keybindings.md) is
generated by `ft commands docs`; CI checks that it stays fresh.

## Modals and the status bar

A modal (picker, multi-step form, query bar, confirmation) takes input
precedence over the active tab. When one is up:

- The status bar's right cell shows `modal: <name>` in magenta.
- The center cell shows up to three primary chords from the modal's
  keymap (`confirm`, `cancel`, and one more).
- Toasts (transient feedback messages) override the hint cell.

Every modal has a `cancel` (usually `Esc`) and most have a `confirm`
(usually `Enter`). The modal-specific bindings are in
[docs/keybindings.md](../keybindings.md) under `modal/<name>`.

## Concurrency

The TUI is single-threaded except for two producer threads — the
crossterm input loop and any background workers (today, just the git
sync). Background jobs own their inputs, post a message back on the
event channel, and exit. There is no async runtime, no shared mutable
state behind a lock, and no risk of the UI freezing because something
synchronous decided to take a while.

The visible effect: a `g s` git sync runs in the background, with a
`⟳ sync` indicator in the status bar. You can keep navigating, editing
tasks, opening notes — every other interaction stays responsive.
Conflicts are surfaced as a status-bar message + toast when the
worker finishes.

## Editor handoff

When the TUI hands off to `$EDITOR` (Notes-tab `o`/`c`/`C`/`t`/`p`
chords, the Tasks-tab `Enter`/`e` chords, the section-move flow's
new-target step, …) it uses the strategy in `[editor]`. Inside tmux,
the default `tmux-popup` keeps the TUI drawing behind the editor.
Outside tmux, it suspends to inline. See
[vault-and-config.md](vault-and-config.md#editor-strategy) for the
strategies and config keys.

## Commands and keymaps

Three patterns are worth knowing:

1. **The registry is the source of truth.** `?` overlay, the markdown
   reference, `ft commands list`, and `ft do` all walk the same set
   of `CommandDef`s. Adding a new chord means adding a row to a
   keymap; adding a new action means adding a `CommandDef` to one of
   the static slices.

2. **Headless dispatch is intentionally narrow.** `ft do <name>` only
   runs commands whose work is atomic and selector-driven. Modal-opening
   commands (`graph.create-blank`, `tasks.edit-popup`) are rejected
   with a clear "use the TUI" message. The expansion path is to
   factor `ft-core` ops out of the TUI as they become atomic enough
   to call without ambient state.

3. **Drift is caught by CI.** `ft commands docs --check` compares the
   shipped `docs/keybindings.md` against the registry and fails the
   build if they diverge. Re-run `ft commands docs > docs/keybindings.md`
   after any registry change.

The full Command/Keymap model and how to add new commands is in
[docs/commands.md](../commands.md).

## When to use the TUI vs. the CLI

- The TUI is the right surface for **exploration**: scanning today's
  tasks, walking the graph, planning the day, capturing thoughts.
  Anything where the next action depends on what you just saw.
- The CLI is the right surface for **automation**: cron jobs, shell
  pipelines, scripts that need a specific output format, anything you
  want to compose with `jq` or `xargs`.
- Both run the same `ft-core` operations underneath. A task you
  completed in the TUI is byte-identical on disk to one you completed
  with `ft tasks complete`.
