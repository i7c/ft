## MODIFIED Requirements

### Requirement: Key bindings live in scoped `KeyMap` data, not match arms

Every key binding in the TUI SHALL be expressed as a `(KeyChord, Command)` row in a `KeyMap`. Tabs SHALL declare their bindings via `Tab::keymap()`; modals SHALL declare theirs via `Modal::keymap()`; the App SHALL declare global bindings via `App::global_keymap()`. No `match KeyCode::Char(_)` arms SHALL remain in tab or modal source for actions covered by the registry. The maps returned by these accessors SHALL be the **effective** maps — the static defaults declared in source overlaid with the user's `[keymap]` table from `config.toml` (see `keymap-config`). The static `<NAME>_KEYMAP: LazyLock<KeyMap>` slices remain the authoritative *defaults* and are what `docs/keybindings.md` documents.

#### Scenario: Default binding lookup
- **WHEN** the Graph tab is active, no modal is up, the user has no `[keymap]` table in `config.toml`, and the user presses `c`
- **THEN** the Graph tab's `keymap().lookup(chord)` returns `Some(Command { name: "graph.create-note", … })` — byte-identical to the pre-overlay behaviour

#### Scenario: Override flows through the same lookup path
- **WHEN** the user's `config.toml` contains `[keymap."tab/graph"]` with `"c" = "graph.refresh"` and the Graph tab is active
- **THEN** `keymap().lookup(chord_for_c)` returns `Some(Command { name: "graph.refresh", … })` — the lookup contract is unchanged; only the data source widens

#### Scenario: Unbound chord falls through
- **WHEN** the active context's effective keymap returns `None` for a chord (because the default never bound it, the user unbound it, or the user's override mapped it to no entry)
- **THEN** the chord falls through to the next layer in precedence (modal → tab → global) and may be unhandled at the end

### Requirement: `?` overlay and `docs/keybindings.md` generated from keymaps

The `?` help overlay SHALL render rows by walking the active context's **effective** `KeyMap` and the global effective `KeyMap`. `docs/keybindings.md` SHALL be generated by walking the central `CommandRegistry` and grouping by scope — using the **static default** keymaps so the committed file is stable across users. Both consumers SHALL share one renderer.

#### Scenario: `?` overlay reflects user overrides
- **WHEN** the user has `[keymap."tab/graph"]` with `"R" = "graph.refresh"` and opens `?` on the Graph tab
- **THEN** the overlay shows the override (`R` next to "Refresh") alongside the still-bound default `Ctrl+r`, all derived from the effective keymap

#### Scenario: `docs/keybindings.md` documents defaults, not user overrides
- **WHEN** `ft commands docs --check` is run in CI on a checkout whose `config.toml` defines a user `[keymap]` table
- **THEN** the check passes because the generator reads the static defaults via the registry, not the effective keymap; the committed file stays user-independent

#### Scenario: `ft commands list --effective` shows resolved bindings
- **WHEN** the user runs `ft commands list --effective`
- **THEN** the output reflects the effective per-scope keymaps (defaults + user overrides + unbinds) — the per-user counterpart to the default-facing `ft commands list`
