## Requirements
### Requirement: Key bindings live in scoped `KeyMap` data, not match arms

Every key binding in the TUI SHALL be expressed as a `(KeyChord, Command)` row in a `KeyMap`. Tabs SHALL declare their bindings via `Tab::keymap()`; modals SHALL declare theirs via `Modal::keymap()`; the App SHALL declare global bindings via `App::global_keymap()`. No `match KeyCode::Char(_)` arms SHALL remain in tab or modal source for actions covered by the registry.

#### Scenario: Binding lookup
- **WHEN** the Graph tab is active, no modal is up, and the user presses `c`
- **THEN** the Graph tab's `keymap().lookup(chord)` returns `Some(Command { name: "graph.create-note", … })`

#### Scenario: Unbound chord falls through
- **WHEN** the active context's keymap returns `None` for a chord
- **THEN** the chord falls through to the next layer in precedence (modal → tab → global) and may be unhandled at the end

### Requirement: Keymap input precedence is modal → tab → global

The TUI input pipeline SHALL resolve incoming key events in the order: active modal's keymap (if any), then active tab's keymap, then App global keymap. A chord consumed by an earlier layer SHALL NOT reach a later layer.

#### Scenario: Modal consumes
- **WHEN** the search picker modal is active and the user presses `Enter`
- **THEN** the modal's `dispatch_command("modal.confirm")` runs and the tab's keymap does not see the event

#### Scenario: Modal falls through, tab handles
- **WHEN** the periodic-leader modal is active, the user presses an unbound printable like `z`
- **THEN** the modal returns `NotHandled`, the App closes the modal (leader's "any-other-key" rule), and the tab keymap evaluates `z` on the now-modal-free state

### Requirement: Keymap collision detection at build time

`KeyMap::new().bind(…)` SHALL detect duplicate chords inside the same scope and SHALL panic (or fail a build-time test) on collision. Cross-scope duplicates are not collisions (a global binding and a tab binding can share a chord; precedence resolves which fires).

#### Scenario: Build fails on duplicate chord in one tab
- **WHEN** two `bind` calls in the same `KeyMap` use the same `KeyChord`
- **THEN** the construction panics with a message naming both commands and the chord

#### Scenario: Cross-scope is not a collision
- **WHEN** the App-global keymap binds `q` and the Graph tab also binds `q`
- **THEN** both keymaps build successfully; precedence (tab before global) determines which fires

### Requirement: `?` overlay and `docs/keybindings.md` generated from keymaps

The `?` help overlay SHALL render rows by walking the active context's `KeyMap` and the global `KeyMap`. `docs/keybindings.md` SHALL be generated by walking the central `CommandRegistry` and grouping by scope. Both consumers SHALL share one renderer.

#### Scenario: `?` overlay rows match keymap content
- **WHEN** the user opens `?` on the Graph tab with no modal active
- **THEN** every row corresponds to one `(chord, command)` entry in the Graph tab's `keymap()` plus the global keymap, and no row is hand-curated

#### Scenario: docs file matches generator output
- **WHEN** `ft completions docs --check` is run in CI
- **THEN** the committed `docs/keybindings.md` matches the output that would be generated from the current registry, byte-for-byte

### Requirement: Modal state hint in the status bar

When a modal is active, the status bar SHALL render the modal's name and up to three primary chords selected from the modal's keymap by `CommandDef.is_primary`.

#### Scenario: Section-move modal hint
- **WHEN** the section-move heading-multiselect modal is active
- **THEN** the status bar renders `MOVE  Space:toggle  Enter:confirm  Esc:cancel` (or the strings derived from the bound commands' `CommandDef.description`)

#### Scenario: Empty hint when no modal
- **WHEN** no modal is active
- **THEN** the status-bar modal-hint cell is empty
